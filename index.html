AMP 배경

배경

Google은 검색 결과에서 링크를 클릭하는 사용자 분석 결과에 따르면

  • 로드 시간이 3초이상 걸리면 모바일 사이트 방문 포기 → 약 53%
  • 로드 시간이 1초 ~ 10초 길어지면 이탈률 → 123% 증가

웹 페이지가 느린 경우 사용자들을 확보 할 수 없음

속도의 영향

웹에서 속도는 중요

전자상거래 사이트가 하루에 10만 달러를 벌고 있다면,
1초 지연으로 인해 매년 250만 달러의 매출 손실이 발생할 가능성이 있다. (참고)

페이지 로드 시간이 수익에 직접적인 영향을 미칠 수 있음

웹 페이지가 느린 이유

  • 웹페이지 용량 - 큰 사이즈 이미지, 동영상, 복잡한 코드와 디자인
  • 리소스 로딩 - 리소스가 로드되는 순서
  • HTTP 요청 수 - CSS와 이미지를 적절히 요약, 외부 리소스 더 적게 포함
  • 느린 JavaScript 실행 - 플러그인, SNS 위젯
  • 웹 폰트 - 용량이 매우 크기 때문에 웹 페이지 성능에 큰 영향을 미침
  • 광고 - 지나치게 많은 광고

비즈니스 사례

경쟁사 위협에 대한 대응, 광고 수익을 유지할 필요성에서 비롯

  • 페이스북 Instant Articles, Apple News
  • 소셜을 통한 뉴스 소비가 늘어나면, 검색을 통한 뉴스 소비는 감소
  • 앱 환경에서 뉴스 형태의 콘텐츠에 초점을 맞춘 방식 (폐쇄적)
  • 구글 광고 수익에 대한 위협 (광고에 대한 안 좋은 인식)
  • 사용자들의 광고 차단, 광고를 보지 않고 느린 사이트를 떠난다면 광고 수익 하락

AMP 소개

AMP

페이지 로딩 시간 단축
긴 로딩 시간이 불러오는 사용자 이탈을 막고 더 나은 컨텐츠 소비 경험 제공

  • Accelerated Mobile Pages (AMP)
  • 웹 페이지를 즉시 로드, 컨텐츠를 빠르게 표시하는 것을 목적
  • 2015년 10월 Google에서 시작한 오픈소스 프로젝트
  • 2016년 2월 부터 Google 모바일 검색 결과에 적용 (1초 이내 로드)
  • 초기 AMP는 뉴스 형태의 콘텐츠에 중점 -> 계속 확장 (전자상거래, 이메일용 AMP)

주요 구성 요소

  • AMP HTML 웹 컴포넌트
  • AMP JS 리소스 관리
  • AMP Cache 캐싱 (CDN)

AMP HTML

  • HTML5 기반으로 AMP 콘텐츠를 작성
  • 하지만 빠른 성능을 위해 사용할 수 있는 HTML 태그 제한
  • 대신 AMP 컴포넌트 (사용자 정의 태그) 제공 <img> → <amp-img>
  • 초기 학습 비용이 부과
  • 기획자, 디자이너 커뮤니케이션 비용
  • AMP HTML 구문 유효성 검사 반드시 체크
  • 오류가 있는 경우 Google은 AMP 페이지를 인식하지 못함
  • 출시 전, 배포 이후 인식되고 있는지 확인 필요 (Google Search Console)

AMP JS

  • AMP 페이지에서 사용자 작성 스크립트가 허용되지 않음
  • AMP 페이지에서 사용 할 수 있는 JS 라이브러리가 제공
  • 모든 JS는 비동기 방식으로 실행, 페이지 로딩에 영향을 미치지 않음
  • 페이지 동작과 관련된 부분은 커스텀 AMP요소를 통해 구현
  • Iframe안에 별도로 외부 JS라이브러리를 포함 가능
  • 빠른 렌더링을 위해 엄격한 엄격한 웹 성능 모범 사례를 따름

AMP Cache

  • AMP 기술을 적용한 페이지는 무료로 Google CDN 서버에 캐싱하는 방식으로 운영
  • AMP 페이지도 빠르지만 AMP Cache로 AMP 페이지를 즉시 로드
  • Google 검색 결과에서 링크 클릭시 실제 사이트로 연결시켜주는 것이 아니라
    미리 캐싱한 데이터를 보여줌으로써 로드 시간 개선
  • 오직 유효한 AMP 문서들을 제공하기 위한 프록시 기반 CDN

AMP 성능 개선 방법

  • 모든 JS는 비동기로 처리 Non-blocking (사용자 JS 허용 안됨)
  • 모든 리소스(이미지, 동영상) 크기를 정적으로 지정
  • 리소스의 다운로드를 기다리지 않고 페이지 레이아웃 위치 ​​계산
  • 인라인 CSS만 허용, 크기를 50KB 제한
  • 외부 리소스를 기다리지 않고 바로 렌더링 시작
  • GPU 가속 애니메이션만 실행
  • GPU가 효과를 처리 할 수 없는 경우 브라우저가 대신 계산을 수행 페이지 렌더링 속도 저하
  • 화면에서 보이는 부분, 사용자가 볼 가능성이 있는 리소스 로드 우선순위 지정
  • 백그라운드에서 웹 페이지 사전 렌더링

AMP 사전 렌더링

사용자가 방문 전 AMP 뷰어를 (숨겨진 iframe) 통해
백그라운드에서 AMP 페이지를 다운로드하고 AMP 캐시에 저장

사전 렌더링된 AMP 링크 클릭 시 완전히 로드되고 AMP 뷰어에 즉시 표시 가능

AMP 논란과 비판

  • AMP Cache는 Google이 소유하고 운영 원래 URL이 아니라 대신 Google 도메인에서 제공
  • Google URL을 사용하면 브랜드가 희석
  • 사이트 입장에서는 사용자 정보도 알 수 없고 트래픽도 발생되지 않음
  • 사용자가 구글 사이트를 떠나지 못하게 잡아두겠다는 의미로 해석
  • Google은 검색 결과에서 AMP 페이지를 특별하게 취급
  • 중앙 집중식 JavaScript
  • 모든 AMP 페이지에는 AMP-JS 라이브러리가 포함
  • 라이브러리를 사용할 수 없거나 손상된 경우 모든 AMP 사이트에 문제 발생
  • AMP가 W3C 표준은 아니라는 점 새로운 웹표준 - 유효한 HTML로 간주되는 표준을 수정
  • 실제 AMP 페이지를 Google을 통하지 않고 접속해 보면 다른 웹페이지와 별반 차이가 크게 없음
  • AMP가 해주는 게 아니라 구글이 검색을 가지고 있기에 가능한 부분
  • 브라우저 지원 범위

AMP == SEO

  • 구글은 모바일 페이지 속도 순위 요소에 우선 순위를 정할 것이라고 발표
  • Google은 모바일 우선 순위 색인에서 페이지 속도를 순위 요소로 사용
  • 기존의 검색 알고리즘을 그대로 적용할 것이며 AMP를 적용한 언론과 적용하지 않은 언론 사이의 차별은 존재하지 않을 것이라는 입장
  • Google 검색 결과 페이지에 AMP 번개 아이콘(⚡)으로 식별
  • AMP Top Stories 구글 검색 결과 상단 노출 검색 결과에서도 상위 노출
  • 모바일 검색 결과에서 AMP 페이지는 검색 상단 Top Stories에 나타납니다.

AMP-JS 라이브러리

  • AMP-JS 기본 라이브러리 페이지에 포함
  • 모든 자바스크립트는 비동기식

  • 모든 스크립트 태그는 주 렌더링을 차단하지 않도록 async 속성 추가
  • 커스텀 스크립트 사용하는 경우 커스텀 태그를 지정
  • AMP HTML 태그를 브라우저가 이해하는 HTML로 변환
  • 리소스 로드 우선 순위를 효율적으로 관리
  • 리소스를 로딩하기 전 모든 요소의 레이아웃을 미리 계산
  • 유효성 검사 기능 포함

AMP 레이아웃 시스템

페이지 레이아웃을 가능한 빠르게 파악하고, DOM 리플로우를 줄이기 위한 핵심 요소

  • 외부 리소스가 로드되기 전, 각 요소의 레이아웃과 크기를 효율적으로 계산
  • layout, width, height, sizes, heights 속성을 통해
    요소의 크기 및 레이아웃 조정
  • CSS 제한(50KB)을 고려할 때 레이아웃 속성은 매우 유용

layout 속성

layout 속성을 추가하여 구성요소의 레이아웃을 지정
